### Finding 🐛 BUG-1: OTel telemetry noise from expected-failure probes

**Severity:** MEDIUM (operational, not user-facing)

**Description:** `src/lib/server/hardware/detection/network-detector.ts` probes three remote services (Kismet @ :2501, OpenWebRX @ :8073, HackRF API @ :8092) on every hardware scan. When the services are down, `fetch()` throws `ECONNREFUSED`, which the function catches and returns `[]`. The application handles this correctly — but OpenTelemetry auto-instrumentation captures the fetch error BEFORE the catch, so every probe creates an error-tagged span.

**Impact:**

- 100% error rate on argos-v1 telemetry → masks real errors in dashboards
- Breaks alerting thresholds set on error-rate metrics
- Adds noise to Sentry if OTel→Sentry bridge is wired

**Evidence:**

- 50/50 sampled traces are these 3 ECONNREFUSED probes
- Source: `src/lib/server/hardware/detection/network-detector.ts:139`, `:164`, `:187` (three `fetch()` calls with `AbortSignal.timeout(2000)`)
- Catch is at lines `:153`, `:178`, `:206` (each branch returns `[]` on failure)

**Suggested fix options (one PR per option, user picks):**

1. **Suppress span error tag** — wrap each probe with a custom span that sets `otel.status_code = OK` even on caught failure. Most explicit.
2. **Add `recordException: false`** to the fetch instrumentation for these specific URLs via OTel config.
3. **Change probe semantics** — use HEAD request to a cheap endpoint OR a connect-only check via `net.connect` with short timeout, less likely to bubble error.
4. **Span attribute** — keep error tag but add `argos.expected_failure = true` so Jaeger dashboards can filter.

**Recommended:** Option 1 (wrap-with-span). Most explicit, doesn't change network behavior, makes intent clear in code.

**Reproducing test plan:** Run argos-v1 with at least one of {Kismet, OpenWebRX, HackRF API} unavailable → inspect Jaeger for `error: true` tagged spans on `network-detector.ts` probes → assert ≤ 0 after fix.

**Severity:** MEDIUM (input-validation gap on attack-relevant data path)

**Description:** `src/lib/schemas/dragonsync.ts:13-50` defines `DragonSyncDroneSchema` (validates RemoteID drone telemetry from upstream DragonSync). **Twelve numeric fields lack `.min()/.max()` bounds**, including position + heading + RSSI:

```ts
pilot_lat: z.number().default(0); // L31 — should be [-90, 90]
pilot_lon: z.number().default(0); // L32 — should be [-180, 180]
home_lat: z.number().default(0); // L33 — should be [-90, 90]

freq: z.number().nullable().default(null); // L37 — should be [1, 6000] MHz

```

**Impact:**

- Attacker controlling upstream RemoteID broadcast (Bluetooth/WiFi-Aware spoofing — there's a literature on this) could emit drone telemetry with `lat: 999, lon: -50000, alt: 1e30, direction: -15`. Schema accepts it.
- Bad lat/lon propagates to map UI (NaN coordinates render at map (0,0) or worse, crash leaflet/maplibre layers — see ARGOS-4 from Sentry as **possible related null-deref** on dashboard `.reduce()` of values that should have been validated).
- CoT (Cursor-on-Target) broadcaster would forward garbage to TAK clients.
- Signal correlation engine could match drone events to RF signals at impossible distances.

**Suggested fix:** Add bounds to all numeric fields per their physical meaning. Use shared schemas from `rf-propagation.ts` pattern (`LatSchema`, `LonSchema`, etc.) — extract to `schemas/common.ts` if not already there.
